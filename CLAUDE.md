# CLAUDE.md

## 目的
このドキュメントは、Claude Code が `report-failed-services` リポジトリで作業を行う際の方針、ルール、およびコンテキストを定義するものです。

## 判断記録のルール
判断を行う際は、以下の項目を記録してください：
1. **判断内容の要約**: 何を決定したか
2. **検討した代替案**: 他にどのような選択肢があったか
3. **採用しなかった案とその理由**: なぜ他の選択肢を選ばなかったか
4. **前提条件・仮定・不確実性**: 判断の根拠となった前提や、不明確な点
5. **他エージェントによるレビュー可否**: Codex などによるレビューが必要か

## プロジェクト概要
- **目的**: Systemd で Failed 状態となったサービスを検知し、Discord に通知する。復帰時にも通知を行う。
- **主な機能**:
  - `systemctl` コマンドによるサービス状態の監視
  - Discord Bot API を使用した通知送信
  - 通知済みサービスの管理 (`notified_ids.json`)
  - 通知除外設定 (`ignore_services.json`)

## 重要ルール
- **会話言語**: 日本語
- **コミット規約**: Conventional Commits (`<type>(<scope>): <description>`)、description は日本語
- **コメント言語**: 日本語
- **エラーメッセージ**: 英語
- **表記**: 日本語と英数字の間に半角スペースを入れる

## 環境のルール
- **ブランチ命名**: Conventional Branch に従う (`feat/`, `fix/` など)
- **リポジトリ調査**: 必要に応じてテンポラリディレクトリを使用する
- **Renovate**: Renovate の PR には直接コミットせず、必要なら別途対応する

## コード改修時のルール
- **エラーメッセージ**: 絵文字を使用する場合、メッセージ全体で統一する
- **ドキュメント**: 関数やモジュールには日本語で docstring を記述・更新する
- **フォーマット**: `flake8` に準拠する

## 相談ルール
- **Codex CLI**: 実装の詳細レビュー、設計の整合性確認に使用する
- **Gemini CLI**: 外部仕様の調査、最新情報の確認に使用する
- **指摘対応**: エージェントからの指摘は無視せず、適切に対応または議論する

## 開発コマンド
```bash
# 依存関係のインストール
pip install -U -r requirements.txt

# Lint (flake8)
flake8 . --count --select=E1,E2,E3,E4,E7,E9,W1,W2,W3,W4,W5,F63,F7,F82 --show-source --statistics

# 実行 (ローカル)
python3 -m src
```

## アーキテクチャと主要ファイル
- **アーキテクチャ**:
  - エントリーポイント (`src/__main__.py`) が `systemctl` を実行し、結果を解析。
  - `src/__init__.py` がユーティリティ (ログ、Discord 送信、状態管理) を提供。
  - `src/config.py` が環境変数をロード。
- **主要ファイル**:
  - `src/__main__.py`: メインロジック
  - `src/__init__.py`: 共通関数 (`send_to_discord`, `load_notified_ids` 等)
  - `src/config.py`: 設定読み込み
  - `SystemdFiles/`: Systemd ユニットファイルとインストーラー

## 実装パターン
- **状態管理**: `notified_ids.json` ファイルを使用して、通知済みのサービス ID を永続化する。
- **設定管理**: 環境変数 (`.env`) と JSON ファイル (`ignore_services.json`) を併用する。

## テスト
- **方針**: 現状、自動化された単体テストは存在しない。
- **確認方法**: 実機での動作確認、またはモックを使用した手動確認を行う。

## 作業チェックリスト

### 新規改修時
1. プロジェクトの構造と既存の実装を理解する
2. 適切な作業ブランチを作成する (`feat/xxx` や `fix/xxx`)
3. 最新のリモートブランチから派生していることを確認する
4. `requirements.txt` の依存関係がインストールされていることを確認する

### コミット・プッシュ前
1. コミットメッセージが Conventional Commits に従っているか確認する
2. `.env` や `logs/` などの機密情報や不要ファイルが含まれていないか確認する
3. `flake8` を実行し、エラーがないことを確認する
4. 実装した機能が期待通り動作することを確認する

### PR 作成前
1. PR 作成の必要性を再確認する
2. コンフリクトが発生しそうな箇所がないか確認する

### PR 作成後
1. CI (`ci.yml`) がパスすることを確認する
2. PR の説明文を日本語で、変更内容がわかるように詳細に記述する
3. レビュー指摘があれば対応する
